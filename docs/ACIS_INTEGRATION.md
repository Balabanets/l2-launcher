# Задача: пускать в игру только игроков, прошедших проверку через лаунчер

Ты в исходниках игрового сервера **aCis 409 (Lineage 2 Interlude, Java)**. Нужно интегрировать
серверную проверку авторизации (Слой 2). Клиентская часть (лаунчер) и бэкенд (сайт) УЖЕ ГОТОВЫ и
работают — твоя задача только дёрнуть проверку при логине и отклонять неавторизованных.

## Как уже работает связка (НЕ переделывать)
- Лаунчер перед запуском игры локально хеширует критичные файлы клиента (`system/*.dll`,
  `*.exe`, `*.int`, `*.dat`) и сверяет с подписанным (Ed25519) манифестом.
- Затем шлёт на бэкенд реальные хеши + одноразовый nonce + HMAC. Бэкенд проверяет подпись
  манифеста, сверяет хеши, валидирует HMAC и **авторизует публичный IP игрока** на короткое
  окно (по умолчанию 120 с, до 4 логинов с одного IP — под NAT).
- Игровой коннект к aCis идёт НАПРЯМУЮ (мимо Cloudflare), поэтому сокет видит тот же
  публичный IP, что и бэкенд видел у HTTP-авторизации. IP совпадают.

## Готовый эндпоинт бэкенда (его и вызывать)
```
GET https://l2.balabanets.uk/api/launcher/check?ip=<IP_игрока>&consume=1
Header: X-Server-Token: <ACIS_SHARED_TOKEN>
→ 200 {"authorized": true|false}     (403 если токен неверный)
```
- `ip` — публичный IP подключающегося игрока (remote address сокета логина).
- `consume=1` — списать одно «окно логина» (одноразовость на вход). Для «мягкой» проверки
  без списания используй `consume=0`.
- `X-Server-Token` — общий секрет `ACIS_SHARED_TOKEN` (тебе дадут; он же в `.env` бэкенда).

## Что реализовать в aCis (Java)
1. Найди точку входа игрока, где уже известен его IP и где можно отклонить вход. Ориентиры:
   `LoginController` / обработчик `RequestAuthLogin` (LoginServer) — там есть IP и аккаунт; либо
   момент `EnterWorld` / выбора персонажа на GameServer (там тоже доступен IP клиента).
   Точные классы уточни по ревизии 409.
2. На этом этапе сделай HTTPS GET к `/api/launcher/check` с IP игрока и заголовком
   `X-Server-Token`. Таймаут 3–5 с.
3. Политика (выбери и задокументируй):
   - **fail-closed (рекомендуется для защиты):** `authorized=false` ИЛИ бэкенд недоступен →
     отказ во входе с сообщением «Запустите игру через лаунчер».
   - **fail-open:** при недоступности бэкенда пускать (меньше защита, выше доступность).
4. Конфиг: вынеси `ACIS_SHARED_TOKEN`, базовый URL и политику в конфиг сервера (НЕ хардкодить
   секрет в коде/гите). Кэшируй результат на пару секунд, чтобы не дёргать бэкенд на каждый пакет.
5. Логирование: фиксируй отказы (IP, время) для диагностики, но НЕ логируй секрет.

## Критерии приёмки
1. Игрок, запустивший игру через лаунчер с целыми файлами, входит нормально.
2. Прямой запуск `l2.exe` без лаунчера (IP не авторизован) → вход отклонён.
3. Подменённые критичные файлы → лаунчер не авторизует IP → вход отклонён.
4. Неверный/отсутствующий `X-Server-Token` → бэкенд отвечает 403 (проверь, что aCis это
   корректно трактует как «нет авторизации»).
5. Секрет только в конфиге/окружении, не в репозитории.

## Честная модель угроз (зафиксируй в README, без иллюзий)
Это поднимает барьер, но не даёт абсолюта на закрытом легаси-клиенте:
- Игрок может запустить лаунчер (авторизовать IP), а затем за окно 120 с запустить
  модифицированный клиент с того же IP. Полностью это не закрыть без модификации клиента.
- HMAC-секрет вшит в лаунчер и извлекаем реверсом — это «мягкое» доказательство, отсекающее
  массовый/казуальный обход, а не криптогарантия.
- IP-привязка слаба при общем NAT (несколько игроков за одним IP).
Если нужна жёсткая привязка к аккаунту — добавь логин в лаунчере и токен, привязанный к
account, который aCis сверяет (потребует доработки и лаунчера, и доставки токена).

## ВАЖНО: про ботов (Adrenaline, L2Walker, пакетные)
Проверка файлов и эта авторизация **НЕ останавливают ингейм-ботов** — они читают/пишут память
запущенного процесса или работают пакетным прокси/эмулятором, файлы не трогая. Реальная защита
от ботов — **серверные поведенческие эвристики в aCis** (отдельная задача):
- анти-спид/анти-телепорт (валидация перемещений на сервере);
- лимиты действий в единицу времени (атаки/подбор/крафт), детект «идеальных» интервалов;
- детект бесконечного фарма/гринда без перерывов, аномалий маршрутов;
- периодическая NPC/чат-капча, особенно для новых аккаунтов;
- эвристики массового создания аккаунтов с одного IP.
Реализуется на стороне Java без касания клиента — это основной инструмент против ботов.
Опционально: проверка списка процессов лаунчером (низкая эффективность, легко обходится) —
не полагаться как на основную меру.

## Лимит окон и «по железу» (HWID) — что где enforced

- **Гейт (этот промпт) — самое важное.** Пока aCis не зовёт `/api/launcher/check`, любой
  запуск проходит (можно играть из папки мимо лаунчера). Гейт делает лаунчер обязательным.
- **Лимит «N окон на ПК» — это уже per-PC и стоит в лаунчере** (он считает локальные `l2.exe`,
  v0.4.6+). Любой лаунчер на ПК видит все окна → 4-е не поднять. Это per-PC по природе (не по IP).
  aCis НЕ нужно считать окна — достаточно гейта, чтобы мимо лаунчера не зайти.
- **HWID-баны (по железу) — на бэкенде.** Лаунчер шлёт SHA-256 от MachineGuid (аутентифицировано
  HMAC) в `/api/launcher/authorize`. Забаненное железо (`LAUNCHER_BANNED_HWIDS` в env l2site, позже
  админка) не авторизуется → гейт не пускает этот ПК. То есть бан «по железу» работает через тот же
  гейт — **aCis ничего дополнительно не делает**.
- **Честно:** легаси-клиент не передаёт HWID на игровой сервер, поэтому *чисто серверного* per-PC
  нет; enforcement идёт через обязательный лаунчер. Неспуфабельная привязка к железу требует
  анти-чита/драйвера (отдельно). По IP лимит НЕ делаем (общий NAT у разных людей).
- **Лимит окон** регулируется `/api/launcher/limits` (`max_clients`, env `LAUNCHER_MAX_CLIENTS`).

## Параметры для подстановки (выдать вместе с промптом)
- `ACIS_SHARED_TOKEN` — из `~/.config/l2-launcher/keys/secrets.env` на сервере (НЕ в git).
- URL проверки: `https://l2.balabanets.uk/api/launcher/check`.
- Окно авторизации/лимит логинов на IP настраиваются в `l2site/src/lib/launcher-auth.ts`
  (`AUTH_TTL_MS`, `AUTH_LOGINS`).
